# 会議の終わりが劇的に変わる！議事録からTODOを自動生成するAIツールを作ってみた

会議、皆さん好きですか？ 私は正直、あまり得意ではありませんでした。特に会議後のタスク整理。議事録作成はまだしも、そこから「誰が、何を、いつまでに」やるべきかを洗い出し、プロジェクト管理ツールに登録する作業が、毎回苦痛でなりませんでした。

「この作業、AIに任せられたら、どれだけ楽になるだろう…？」

そんな思いから、私は議事録からTODOを自動生成するAIツールを開発することにしました。この記事では、私がこのツールをどう設計し、どんな壁にぶつかり、最終的に何を得たのかを、包み隠さずお話ししたいと思います。
「会議後のあの面倒な作業、なんとかしたい！」と感じているあなたに、きっと役立つはずです。

## 1. なぜ作ったか（僕の会議ストレスをAIで解決したかった）

### 終わらない会議後の「宿題」
私の仕事では、毎日複数の会議に参加します。議題の決定、情報共有、意思決定…会議そのものは重要です。しかし、会議が終わった後の「TODOの洗い出し」と「タスク管理ツールへの登録」が、私の長年のストレス源でした。

議事録はきちんと残します。でも、その議事録を読み返し、

*   「あれ、この発言、誰がやるべきタスクだっけ？」
*   「これはいつまでに終わらせるんだっけ？」
*   「この議論は、具体的なタスクに落とし込めるだろうか？」

といった思考を巡らせながら、手作業でタスクを抽出していくのが、本当に手間でした。時には、抽出漏れで後から慌てたり、タスクの表現が曖昧で結局自分で整理し直したりすることも。

会議が終わっても、本当の意味での「解放感」は得られず、むしろ「これからTODO整理が待っている…」というプレッシャーを感じていたのです。

### AIなら解決できるはず！
ChatGPTの登場以来、私はその可能性に魅了されていました。自然言語処理能力が飛躍的に向上した今なら、きっと議事録という unstructured data から、structured な TODO リストを生成できるはずだと確信しました。

「AIにこの面倒な作業を任せられれば、私はもっと創造的な仕事に集中できる。会議後の開放感を、本当の意味で味わえるようになる！」

この強いモチベーションが、私の開発の原動力となりました。

## 2. 設計と技術選定（最適なLLMとフレームワークを求めて）

### どのLLMを使うか？ 精度とコストのバランス
まず、最も重要なのはどの大規模言語モデル（LLM）を選ぶかでした。

*   **GPT-3.5-turbo**: 処理速度が速く、APIコストも比較的安い。しかし、複雑な指示への追従性や出力の安定性に課題がある可能性も。
*   **GPT-4**: 圧倒的な精度と複雑な指示への対応力。しかし、APIコストが高く、処理速度もGPT-3.5-turboに比べて遅い。

開発初期段階で、いきなり高コストなGPT-4を使うのは避けたいと考えました。まずはコストパフォーマンスに優れるGPT-3.5-turboで基本的な機能を実現し、もし精度が足りない場合にGPT-4への切り替えや併用を検討する方針にしました。
議事録からのTODO抽出は、そこまで複雑な論理推論を必要とせず、むしろ「テキストの理解」と「構造化」がメインなので、GPT-3.5-turboでも十分な精度が出せるのではないか、と予想しました。

### プロンプト管理と拡張性。LangChain一択だった
次に、LLMと連携するためのフレームワークです。

*   **自前でAPIを叩く**: 最もシンプルな方法ですが、プロンプトの管理、エラーハンドリング、後々の機能拡張（例えば、議事録の要約を追加したり、外部ツールと連携したり）を考えると、コードが肥大化しがちです。
*   **LangChain**: LLMアプリケーション開発のための強力なフレームワーク。プロンプトテンプレート、チェイン、エージェント、出力パーサーなど、LLMと連携する上で必要なコンポーネントが豊富に用意されています。

私は迷わず **LangChain** を選びました。その理由は以下の通りです。

1.  **プロンプト管理の容易さ**: 複雑なプロンプトをテンプレートとして管理でき、変数埋め込みも簡単。
2.  **チェイン機能**: 複数のLLM呼び出しやその他の処理ステップを効率的に連結できる。今回の用途であれば、「議事録入力 → TODO抽出 → 出力整形」といったパイプラインが組みやすい。
3.  **出力パーサー**: LLMの出力は時に不安定なため、特定のフォーマット（JSONなど）に強制したり、パースエラーをハンドリングしたりする機能は必須だと考えました。
4.  **将来的な拡張性**: 将来的に音声入力からの議事録生成や、既存のタスク管理ツール（Jira, Trelloなど）への自動登録といった機能を追加する際に、LangChainのエージェント機能などが役立つと見込みました。

このような検討を経て、「**GPT-3.5-turbo をコアに、LangChain で開発を進める**」という方針が固まりました。

## 3. 実装のポイント（プロンプトとLangChainで知恵を絞る）

### プロンプトエンジニアリングが命！具体的な指示の重要性
LLMの性能を最大限に引き出すには、適切なプロンプトが必要です。特に、ただ「TODOを抽出して」と言うだけでは、曖昧な出力になりがちです。

私が意識したのは、以下の点です。

1.  **明確な役割付与**: 「あなたは優秀なプロジェクトマネージャーアシスタントです。」のように、LLMに役割を与え、その役割になりきってタスクを抽出させる。
2.  **具体的な抽出条件**: 「誰が (担当者)」「何を (タスク内容)」「いつまでに (期限)」「状態 (未着手、進行中など)」を必ず抽出するよう指示。
3.  **出力フォーマットの厳格な指定**: 今回はJSON形式で出力させることにしました。これにより、後続の処理が非常に楽になります。PydanticモデルとFunction Callingを組み合わせることで、この安定性が格段に向上しました。
4.  **例示（Few-shot prompting）**: 可能であれば、良い抽出例をいくつかプロンプトに含めることで、LLMの理解を深め、出力の精度を上げることができます。

### LangChainでスマートに実装
LangChainを使うことで、プロンプトの定義、LLMの呼び出し、出力のパースまでを非常に少ないコード量で実現できました。

以下に、核となるコードの一部を紹介します。

```python
import os
from typing import List, Optional
from datetime import date
from pydantic import BaseModel, Field

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.pydantic_v1 import BaseModel, Field
from langchain_core.output_parsers import PydanticOutputParser

# 環境変数にAPIキーを設定
# os.environ["OPENAI_API_KEY"] = "YOUR_OPENAI_API_KEY"

# 1. TODOの構造をPydanticで定義
class Todo(BaseModel):
    task: str = Field(description="具体的なタスク内容。何をするのかを明確に記述してください。")
    assignee: str = Field(description="タスクの担当者名。議事録から明確に読み取れる場合のみ記述してください。")
    due_date: Optional[date] = Field(default=None, description="タスクの期日。議事録から具体的な日付が読み取れる場合のみ記述してください。YYYY-MM-DD形式。")
    status: str = Field(default="未着手", description="タスクの現在の状態（例: 未着手、進行中、完了待ちなど）。")
    context: str = Field(description="タスクが生まれた議事録内の文脈や背景。")

class TodoList(BaseModel):
    todos: List[Todo] = Field(description="会議議事録から抽出されたTODOアイテムのリスト。")

# 2. LLMのインスタンス化
llm = ChatOpenAI(model="gpt-3.5-turbo-0125", temperature=0) # 温度は0で安定した出力を目指す

# 3. プロンプトテンプレートの定義
# Function Calling を使用する場合、出力形式はスキーマに任せるため、詳細なJSON指示は不要
# むしろ、どのような情報を抽出してほしいかを明確に伝えることが重要。
prompt = ChatPromptTemplate.from_messages([
    ("system", """
    あなたは優秀なプロジェクトマネージャーアシスタントです。
    提供された会議議事録から、具体的なTODOタスクを抽出してください。
    
    以下の点を厳守してください:
    - 各タスクは「誰が」「何を」「いつまでに」が明確になっている必要があります。
    - 期限が不明な場合は、due_dateはnullにしてください。
    - 担当者が不明な場合は、assigneeは「未定」またはnullにしてください。
    - 抽出したタスクの具体的な内容と、それが議事録のどの部分から抽出されたかを 'context' フィールドに記述してください。
    - 抽象的な議論ではなく、具体的な行動を伴うタスクのみを抽出してください。
    - 出力は指定されたJSONスキーマに厳密に従ってください。
    """),
    ("human", "以下の議事録からTODOを抽出してください。\n\n{minutes_of_meeting}")
])

# 4. LangChain Runnable の構築 (Function Calling を利用)
# with_structured_output を使うことで、Pydanticモデルに沿ったJSON出力を強制できる
todo_extraction_runnable = prompt | llm.with_structured_output(TodoList)

# 5. 実行例
minutes = """
## 2024年3月15日 定例会議議事録

**参加者:** 山田、佐藤、田中

### 議題1: 新規Webサイトのリニューアルについて

*   山田さんから、既存サイトのアクセス解析結果が共有されました。特にモバイルからのアクセスが多いことが判明。
*   佐藤さん: 「モバイル最適化は最優先で進めましょう。デザインチームには、来週金曜日（3月22日）までにプロトタイプを提出してもらうよう依頼します。」
*   田中さん: 「コンテンツの見直しも必要ですね。既存ページの棚卸しと、新しいページの企画を私が担当します。来月末（4月30日）までには草案を作成します。」
*   山田さん: 「プロトタイプが上がってきたら、社内でのフィードバック会を企画します。日程調整は私が行います。」

### 議題2: 営業資料のアップデート

*   現行の営業資料が古くなっているため、最新の製品情報に更新する必要があるとの意見が出ました。
*   佐藤さん: 「この件は来週の営業会議で詳細を詰めましょう。一旦、私がたたき台を用意しておきます。」

以上
"""

# タスク抽出
try:
    result: TodoList = todo_extraction_runnable.invoke({"minutes_of_meeting": minutes})
    
    print("--- 抽出されたTODOリスト ---")
    for i, todo in enumerate(result.todos):
        print(f"TODO {i+1}:")
        print(f"  タスク: {todo.task}")
        print(f"  担当者: {todo.assignee}")
        print(f"  期日: {todo.due_date if todo.due_date else 'なし'}")
        print(f"  状態: {todo.status}")
        print(f"  文脈: {todo.context[:50]}...") # 長すぎるので冒頭50文字
        print("-" * 20)

except Exception as e:
    print(f"エラーが発生しました: {e}")

```

このコードでは、`llm.with_structured_output(TodoList)` を使うことで、LLMが `TodoList` Pydanticモデルに厳密に従ったJSONを生成するように強制しています。これはGPT-3.5-turbo-0125以降で利用できるFunction Callingの機能で、出力の安定性を格段に高めてくれます。

## 4. ハマったこと（LLMとの格闘と賢い付き合い方）

### 出力フォーマットの不安定さとの戦い
**ハマったこと:**
開発初期、最も頭を悩ませたのがLLMの出力フォーマットの不安定さでした。JSON形式で出力するように指示しても、たまに余計なテキストが混ざったり、引用符の付け方がおかしかったり、時には全くJSONにならないこともありました。特に議事録が長くなると、この傾向が顕著でした。

「せっかくタスクは抽出できていても、JSONとしてパースできないと次の処理に進めない…！」

私は何度もプロンプトを修正し、「JSON形式で出力してください。他のテキストは一切含めないでください」と厳しく指示しましたが、完全には解決しませんでした。

**解決策:**
この問題は、**LangChainの `with_structured_output` 機能（裏ではOpenAIのFunction Calling）とPydanticモデルを組み合わせる**ことで劇的に改善されました。

*   `PydanticOutputParser` を利用し、LLMに特定のPydanticモデルの形式で出力させる。
*   `llm.with_structured_output(PydanticModel)` を使うことで、LLMはモデル定義を解釈し、そのスキーマに沿ったJSONを生成しようとします。これはプロンプトで指示するよりもはるかに強力です。
*   これにより、LLMは出力形式を気にすることなく、タスク抽出という「中身」に集中できるようになりました。結果として、出力の安定性が飛躍的に向上し、パースエラーがほとんど発生しなくなりました。

### タスク抽出の粒度と曖昧さ
**ハマったこと:**
抽出されるTODOの粒度がバラバラだったり、議事録の抽象的な議論までタスクとして抽出してしまうことがありました。

例えば、「市場の動向を調査する」という発言があったとして、これは確かにTODOですが、「誰が」「何を」「いつまでに」が不明確なままだと、実用的なタスクにはなりません。また、「来週、〇〇について議論しましょう」といった単なる議題設定の発言までタスクとして抽出してしまうこともありました。

**解決策:**
プロンプトに「**具体的な行動を伴うタスクのみを抽出する**」という指示を強調し、「**誰が、何を、いつまでに、が明確なタスク**」という条件を強く入れました。また、「議論、検討、報告のみ、など具体的な行動を伴わないものはタスクとして抽出しない」といったネガティブプロンプトも追加しました。

そして、`assignee` や `due_date` が不明な場合は `null` にすることや、`context` フィールドに元の議事録の文脈を含めることで、後から人間が確認・修正しやすくしました。これにより、不明確なタスクでも元の情報に遡って判断できるようになったため、ツールの実用性が向上しました。

### トークン制限とコスト
**ハマったこと:**
非常に長い議事録（数千文字以上）をLLMに投げると、トークン制限に引っかかったり、APIコストが高騰したりする問題がありました。特にGPT-4を使うと顕著です。

**解決策:**
今回はGPT-3.5-turboをメインにすることで、ある程度の長さの議事録に対応できました。しかし、根本的な解決策としては、

*   **前処理での要約**: LLMに渡す前に、議事録を自動で要約するステップを挟むことで、トークン数を削減し、重要な情報に絞って抽出させる。
*   **チャンク分割**: 議事録を意味のある塊（チャンク）に分割し、それぞれからTODOを抽出し、後で統合する。
*   **モデルの使い分け**: 最初のTODO抽出はGPT-3.5-turboで行い、より複雑な推論や洗練が必要な場合にのみGPT-4を使うといった、ハイブリッドなアプローチも検討しています。

現在はまだ前処理までは実装していませんが、今後の改善点として考えています。

## 5. 成果と学び（AIが僕の仕事をどう変えたか、そして未来へ）

### 劇的に変わった会議後の僕
このツールを導入してから、私の会議後のワークフローは劇的に変化しました。

*   **TODO抽出時間の短縮**: 以前は15分〜30分かかっていたTODO整理が、数秒で完了するようになりました。LLMが生成したTODOリストを最終確認するだけで済むため、圧倒的な時間の節約です。
*   **抜け漏れの減少**: 人間が見落としがちな細かなタスクや、過去の議論から派生するタスクなどもAIが自動で拾い上げてくれるため、タスクの抜け漏れが大幅に減りました。
*   **精神的負担の軽減**: 「面倒な作業が待っている」という会議後のプレッシャーから解放され、本当に「会議が終わった！」という達成感を感じられるようになりました。これは何よりの成果です。
*   **チーム連携の円滑化**: 抽出されたTODOリストは明確な形式で出力されるため、他のチームメンバーとの共有も非常にスムーズになりました。

### AIと向き合って得た学び
AIツール開発を通じて、私は多くの学びを得ました。

*   **LLMは魔法ではない、しかし強力なツール**: 何でもできるわけではありませんが、適切な指示と工夫を凝らせば、人間では効率的に行えない作業を自動化し、生産性を劇的に向上させる力があります。
*   **プロンプトエンジニアリングは「対話術」**: LLMとの対話は、まるで新人教育のようです。具体的な指示、期待するアウトプットの明示、そして失敗からの改善の繰り返しが、良い結果を生みます。
*   **フレームワークの重要性**: LangChainのようなフレームワークは、LLM開発の複雑さを吸収し、開発者が「何をしたいか」という本質的な部分に集中できるようにしてくれます。特に、出力の構造化は実用的なアプリケーションを作る上で不可欠です。

### 次にやること、そして未来へ
今回作成したツールはあくまで第一歩です。今後は以下のような改善や拡張を検討しています。

1.  **音声議事録からの直接入力**: 現在はテキストをコピー＆ペーストしていますが、Zoomなどの会議録音から自動生成されたテキストを直接入力できるようにしたいです。これにより、さらに手間が削減されます。
2.  **RAG (Retrieval-Augmented Generation) を用いた既存システムとの連携**: 既存のタスク管理ツール（Jira, Asana, Notionなど）に自動でタスクを登録できるようにする。これにより、TODO抽出から登録までの一連の流れを完全に自動化できます。
3.  **UIの作成**: より直感的に使えるWebアプリケーションとして提供し、チーム全体で使えるようにしたいです。
4.  **多言語対応**: グローバルなチームでの利用も視野に入れ、多言語の議事録からの抽出に対応できるようにしたいです。
5.  **より洗練されたプロンプト**: 特定のドメイン知識（例えば、ソフトウェア開発、マーケティングなど）を持つタスクをより正確に抽出できるように、Few-shot Exampleを拡充したり、場合によってはFine-tuningも検討します。

AIは、私たちの仕事を奪うものではなく、私たちを「面倒な作業」から解放し、より創造的で価値のある仕事に集中させてくれる、最高のパートナーだと感じています。会議後のTODO整理に悩む皆さん、ぜひAIの力を活用してみてはいかがでしょうか。あなたの会議体験も、きっと劇的に変わるはずです！
