4ヶ月で153PR、本番稼働。単独エンジニアの実測データです。

これは「AIを使えば速くなる」という話ではありません。
**どう使えば速くなるか**の、具体的な設計と結果の話です。

---

## 背景

2026年3月、中古車輸出会社の業務管理システムを唯一のエンジニアとして構築し始めました。

スタックは：Go + React + PostgreSQL + GCP（Cloud Run / Cloud SQL / Pub/Sub / Vertex AI）+ Kubernetes + Argo CD + OpenTelemetry。

外注すれば8〜15百万円（年間推定）のコスト。それを75営業日で本番カットオーバーさせました。

---

## 開発プロセスの設計

「AIを使う」だけではうまくいきません。
どの判断を自分が持ち、どの作業をAIに任せるかを明確にしないと、レビューもできない、責任も持てない。

私が設計した分担は以下です。

### 人間がやること

- **要件定義**: 現場を観察し、何を解決すべきかを判断する
- **受入基準の設定**: 「これが動けばOK」の基準を書く
- **本番判断**: デプロイするかどうか、リスクを取るかどうか
- **設計のレビュー**: AIが出した設計に穴がないか確認する

### AIに任せること

- 実装（コード生成・リファクタリング）
- テスト作成
- PR本文・コミットメッセージ
- デバッグの候補提示

ポイントは「AIを使う」ではなく「AIが実装し、人間がジャッジする」という構造にしたことです。

**AIが実装を担当することで、私がより多くの時間をレビューと判断に使えた**結果です。

## ワークフローの実例

1つのIssueを例に、実際の流れを見てみます。

### ステップ1: Issue起票（自分）

```
feat: RBAC ロールベースアクセス制御の実装
- admin / editor / viewer の3ロール
- JWT トークンのロールクレームで判定
- requireRole() ミドルウェア
受入基準: GET /tasks は viewer 以上、DELETE は admin のみ
```

### ステップ2: 実装（Claude Code）

Claude Code がコードを生成。
roleMiddleware.js、テストファイル、OpenAPIドキュメントまで一気通貫。

### ステップ3: レビュー（自分）

- ロジックに穴がないか
- テストが実際のユースケースをカバーしているか
- エッジケースが漏れていないか

ここは絶対に省略しない。AIが作ったコードを「動いた」で終わらせない。

### ステップ4: PR作成・マージ

PR本文もAIが生成し、自分がレビューして修正。

## 機能しなかったパターン

最初は「とにかくAIに任せる」スタンスで始めました。

うまくいきませんでした。

コードは動くが、設計が雑になっていく。テストが形だけになる。本番でのエッジケースを見落とす。

「AIが速い」のではなく、「人間の判断が間に合わなくなった」状態でした。

そこで立て直したのが、「**AIは実装層のみ**」という境界設計です。

要件定義・受入基準・設計レビュー・本番判断——この4つは自分が持つ。
実装・テスト作成・ドキュメント生成——これをAIに任せる。

この分担を決めてから、速度と品質が両立するようになりました。

---

## 可視化: テクニカルデットの計測

「速く作る」と「負債を作る」は違います。

私はOpenTelemetry + Grafanaで各コンポーネントのレイテンシを常時計測し、技術的負債をGitHub Issueに数値付きで記録しました。

```
Issue #87: データベースクエリ最適化
- 現状: GET /tasks p95 = 340ms
- 目標: p95 < 100ms
- 原因: N+1クエリ（計測済み）
```

「なんとなく遅い」ではなく「どこが何msで遅い」まで落とせると、優先度がつけられます。

---

## この記事で伝えたかったこと

「AI-native開発」は、AIに全部やらせることではありません。

**人間の判断が必要な場所を明確にして、AIが介入できない壁を設ける**——それが設計です。

153PRという数字は、AIが速かったのではなく、私がジャッジに集中できた75日間の結果です。

受入基準を書ける人、設計の穴を指摘できる人、本番リスクを引き受けられる人——AIが広まるほど、その役割の価値は上がります。
