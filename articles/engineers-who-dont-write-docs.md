「コードを見れば分かる」

ドキュメントを書かない開発者が言う、定番のセリフです。

コードは確かに正確です。動作の事実が書いてある。

でも、**なぜそう実装したか**は、コードには書いてありません。

## コードで分かることと分からないこと

コードを読めば分かること:
- 何をしているか
- どう動いているか

コードを読んでも分からないこと:
- なぜそう実装したか
- どんな選択肢を検討して、なぜこれを選んだか
- どんな制約や前提があるか
- 将来変える可能性があるかどうか

「なぜ」の部分が失われると、後からコードを読む人は間違った修正をするリスクが上がります。

## ドキュメントを書かない本当の理由

「コードを見れば分かる」は、建前であることが多いです。

本当の理由:
- 書くのが面倒
- 書き方が分からない
- 後で書こうと思って忘れる
- 陳腐化することへの懸念（どうせすぐ古くなる）

これらは全部、本音です。そして全部、理解できます。

ドキュメントは書いた瞬間から劣化が始まります。
コードが変われば、ドキュメントとズレが生じる。

## それでも書いた方がいい理由

ドキュメントの価値は、「なぜ」を記録することにあります。

コードが変わっても、「当時なぜそうしたか」は変わりません。
意思決定の記録は、陳腐化しません。

特に価値が高いのは:
- 採用しなかった選択肢とその理由
- 特定のライブラリを使った/使わなかった理由
- パフォーマンス上のトレードオフ
- セキュリティ上の判断

これらをコードのコメントや README に残すだけで、後から読む人（未来の自分を含む）の助けになります。

## 未来の自分への借金

ドキュメントを書かないことは、未来の自分への借金です。

6ヶ月後に同じコードを読んだ時、「なぜこう書いたんだっけ」と悩む時間が発生します。

その時間は、今ドキュメントを書く時間より長いことが多い。

## 最低限のドキュメント

全部を書く必要はありません。

最低限これだけあれば、後から読む人が助かる:

- **README**: 起動方法・依存関係・環境変数
- **ADR（Architecture Decision Record）**: 大きな設計判断とその理由
- **コードコメント**: 非自明な実装の「なぜ」だけ

網羅的なドキュメントより、「なぜ」が書いてある最小限のドキュメントの方が価値があります。

## おわりに

「コードを見れば分かる」は半分正しい。

動作は分かる。意図は分からない。

意図を残すことが、ドキュメントの本質的な役割です。

完璧なドキュメントを目指さなくていい。
「なぜそうしたか」を一行書くだけで、未来の誰かの時間を救えます。
